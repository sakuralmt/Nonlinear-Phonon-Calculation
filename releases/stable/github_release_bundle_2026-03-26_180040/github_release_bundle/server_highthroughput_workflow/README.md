# Server High-Throughput Workflow

This directory packages the server-side `stage2` and `stage3` route.

Default host split:

- `stage1`: `159.226.208.67:33223`
- `stage2/3`: `100.101.235.12`

Cross-machine handoff is contract-based. The bundle does not automate SSH transfer.

Current focus:

1. assess CPU / threading behavior on the target machine
2. run the optimized CHGNet screening path
3. optionally chain CHGNet screening to later QE handoff

## Files

- `bootstrap_server_env.sh`
  - builds a clean `qiyan-ht` conda environment on the server
  - installs the CPU PyTorch wheel, `chgnet`, and editable local `GPTFF`
- `portable_cpu_config.json`
  - portable CPU runtime profile
  - intended to migrate across different CPU machines without hand-editing
- `portable_cpu_small.json`
- `portable_cpu_medium.json`
- `portable_cpu_large.json`
  - optional CPU portability presets for smaller / balanced / larger machines
- `CPU_QUICKSTART_zh.md`
  - short Chinese quickstart for migrating to a new CPU machine
- `assess_chgnet_env.py`
  - measures CHGNet CPU inference behavior on the current machine
  - writes `env_reports/chgnet_runtime_config.json` as a machine-specific fixed config
  - also inspects the full Slurm partition layout
  - writes `env_reports/slurm_cluster_assessment.json`
  - writes `env_reports/slurm_runtime_config.json`
  - writes `env_reports/slurm_submit_defaults.sh`
- `run_server_pipeline.py`
  - one top-level controller for `screening -> QE recheck`
  - defaults to the current working directory as the repo root
- `scheduler.py`
  - tiny scheduler abstraction layer
  - resolves `auto|slurm|local` for the MLFF screening stage
- `stage_contracts.py`
  - defines the handoff manifests between the 3 workflow layers
- `run_modular_pipeline.py`
  - modular runner for `stage1|stage2|stage3|all`
  - supports resume / file handoff across different machines

## Runtime Config Precedence

For the accelerated CHGNet screening path, runtime settings are resolved in this
order:

1. explicit `--runtime-config`
2. explicit `--runtime-profile`
3. `server_highthroughput_workflow/env_reports/chgnet_runtime_config.json`
4. `server_highthroughput_workflow/portable_cpu_config.json`
5. built-in auto CPU heuristics

This means:

- if you benchmark a machine once with `assess_chgnet_env.py`, later runs can
  reuse the fixed machine-specific config automatically;
- if no assessed config exists, the portable CPU profile will still adapt to the
  local CPU topology automatically.

Explicit overrides:

- `--runtime-config <json>`
  - force a specific runtime config file
- `--runtime-profile small|medium|large|default`
  - force one of the bundled portable CPU presets
- `--scheduler auto|slurm|local`
  - choose whether MLFF screening is queued through Slurm or run locally

For Slurm-backed jobs, the runtime now probes the live cluster if the hard-coded
partition or walltime is invalid. The generated shell snippet can also be
`source`d to pin machine-specific defaults:

```bash
source server_highthroughput_workflow/env_reports/slurm_submit_defaults.sh
```

## Expected server assumptions

- `pw.x`, `sbatch`, `squeue`, and `srun` are available
- Intel oneAPI MPI can be initialized with:

```bash
source /opt/intel/oneapi/setvars.sh >/dev/null 2>&1
```

## Typical usage

Run from the repository root on the server:

```bash
bash server_highthroughput_workflow/bootstrap_server_env.sh
~/miniconda3/envs/qiyan-ht/bin/python server_highthroughput_workflow/assess_chgnet_env.py
~/miniconda3/envs/qiyan-ht/bin/python mlff_modepair_workflow/run_pair_screening_optimized.py --backend chgnet --model r2scan
~/miniconda3/envs/qiyan-ht/bin/python server_highthroughput_workflow/run_server_pipeline.py
```

Outputs are grouped under:

```bash
server_highthroughput_workflow/runs/<run_tag>/
```

The controller writes a single high-level summary file:

```bash
server_highthroughput_workflow/runs/<run_tag>/workflow_summary.json
```

The current portable CPU profile ships in:

```bash
server_highthroughput_workflow/portable_cpu_config.json
```

Additional portable profiles ship in:

```bash
server_highthroughput_workflow/portable_cpu_small.json
server_highthroughput_workflow/portable_cpu_medium.json
server_highthroughput_workflow/portable_cpu_large.json
```

If a machine-specific config is generated by `assess_chgnet_env.py`, the
controller and optimized screening script automatically prefer:

```bash
server_highthroughput_workflow/env_reports/chgnet_runtime_config.json
```

Quickstart:

```bash
server_highthroughput_workflow/CPU_QUICKSTART_zh.md
```

If Slurm is unavailable and `--scheduler auto` is used, the controller falls
back to local CHGNet screening and stops after producing the screening ranking.

## Modular 3-Stage Flow

The current workflow can also be run as three explicit layers with file handoff:

1. stage1
   - runs the real phonon frontend and then packages `selected_mode_pairs.json`, `scf.inp`, and pseudos
   - writes `stage1_manifest.json`
2. stage2
   - runs CHGNet screening from the stage1 manifest
   - writes `stage2_manifest.json`
3. stage3
   - runs QE handoff / QE recheck from the stage2 manifest
   - writes `stage3_manifest.json`

Example:

```bash
python server_highthroughput_workflow/run_modular_pipeline.py --stage stage1 --run-root /path/to/run_root
python server_highthroughput_workflow/run_modular_pipeline.py --stage stage2 --run-root /path/to/run_root --runtime-profile medium
python server_highthroughput_workflow/run_modular_pipeline.py --stage stage3 --run-root /path/to/run_root --qe-mode prepare_only --scheduler local
```
